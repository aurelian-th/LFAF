## Theory
While lexical analysis (Lexing) transforms raw text into a linear sequence of tokens, Syntax Analysis (Parsing) gives those tokens structural and semantic meaning. A **Parser** processes the stream of tokens against a defined formal grammar to ensure the text is syntactically valid.

The most common output of a parser is an **Abstract Syntax Tree (AST)**. Unlike a concrete parse tree, which includes every single syntactic detail (like semicolons, parentheses, and whitespaces), an AST abstracts away redundant punctuation and strictly represents the hierarchical logic of the source code. 

Building a compiler front-end typically involves:
1. **Regular expressions (Regex):** Used in modern lexers to efficiently categorize tokens (e.g., matching `\d+` for integers or `\b[a-zA-Z_]+\b` for keywords).
2. **Recursive descent parsing:** A top-down parsing strategy where a set of mutually recursive functions process the token stream, matching the grammar rules left-to-right.

## Implementation description
For this laboratory work, I built a complete compiler front-end for **Isomorph**, a custom diagramming Domain-Specific Language.

### 1. The regex lexer
The manual character-by-character scanner from Lab 3 was entirely refactored. I implemented a `TokenType` enumeration containing the Regex patterns for keywords (using word boundaries `\b`), punctuation, and data literals. The `Lexer` dynamically concatenates these into a single master Regex using named capturing groups (e.g., `(?<IDENTIFIER>[a-zA-Z_]+)`). It uses Java's `Matcher.find()` to iterate through the text, instantly categorizing matches and silently dropping structural whitespace and comments.

### 2. The Abstract Syntax Tree (AST)
I mapped the Isomorph language grammar into hierarchical Object-Oriented nodes encapsulated in an `AST` class:
* `ProgramNode`: The root of the tree.
* `HeaderNode`: Stores diagram metadata (type, theme, resolution).
* `EntityNode`: Represents actors, databases, and participants.
* `RelationshipNode`: Captures directed edges, custom connection types, and property blocks.
* `LayoutNode`: Stores coordinate data.

### 3. The recursive descent parser
The `Parser` consumes the `Token` list generated by the Lexer. It utilizes standard `peek()`, `advance()`, and `consume()` methods to strictly enforce syntax. For example, the `parseHeader()` method expects a `KW_DIAGRAM` token, consumes an `IDENTIFIER` for the name, expects an opening brace, and recursively loops to parse configuration key-value pairs until the closing brace is met. 

## Challenges and difficulties
**1. Java regex named group restrictions:**
While compiling the master Regex pattern, Java threw a fatal `PatternSyntaxException`. It turns out Java's regex engine strictly forbids underscores (`_`) in named capturing groups. Because my enum values were named dynamically (e.g., `KW_DIAGRAM`), the engine crashed on the underscore. 
* *Solution:* I implemented an on-the-fly string formatter in the Lexer that strips underscores from the enum name before injecting it into the compiler (`KW_DIAGRAM` $\to$ `KWDIAGRAM`) and does the same when fetching the matched group.

**2. Parsing complex multi-token structures:**
In the Isomorph DSL, relationship arrows can be simple (`->`) or explicitly typed (`-[dashed]->`). 
* *Solution:* I utilized lookahead token matching inside `parseRelationship()`. If the parser matches a standard `ARROW` token, it sets the type to "solid". If it matches a `MINUS` token, it expects a left bracket, consumes the inner identifier to set the line type, expects a right bracket, and finally expects the `ARROW`.
